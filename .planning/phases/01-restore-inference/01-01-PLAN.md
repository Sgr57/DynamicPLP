**Part B: Chat template auto-injection (per D-03, D-04)**

In `src/lib/adapters/transformersJsAdapter.js`, modify the `load()` method. After the `AutoTokenizer.from_pretrained()` call completes (after line 60 in the current working tree), and BEFORE the model load starts, insert the chat_template detection and fetch logic:

```javascript
// Generic: auto-detect and fetch chat_template for any model that ships it separately (D-03)
if (!this.tokenizer.chat_template) {
  const templateUrl = `https://huggingface.co/${this.modelId}/resolve/main/chat_template.jinja`
  try {
    const response = await fetch(templateUrl)
    if (response.ok) {
      this.tokenizer.chat_template = await response.text()
    } else if (response.status === 404) {
      throw new Error(
        `Modello ${this.modelId}: chat_template mancante (non in tokenizer_config.json ne' come file .jinja)`
      )
    } else {
      throw new Error(
        `Impossibile scaricare chat_template: ${response.status} ${response.statusText}`
      )
    }
  } catch (err) {
    if (err.message.includes('chat_template')) throw err
    throw new Error(`Errore di rete scaricando chat_template: ${err.message}`)
  }
}
```

This code:
- Checks `this.tokenizer.chat_template` after tokenizer load -- generic, works for any model (D-03)
- Fetches from HuggingFace Hub standard `/resolve/main/` URL pattern
- On success: assigns template string to `this.tokenizer.chat_template`
- On 404: throws descriptive error mentioning both possible locations (tokenizer_config.json and .jinja file)
- On other HTTP errors: throws with status code and text
- On network errors: re-throws with descriptive message
- All errors propagate up through the worker's `catch` block, which sends `{type: 'error', message}` to `useModelLoader`, which retries (MAX_RETRIES=2) and then sets `status='error'` (D-04)

Also update the JSDoc comment at the top of the `load()` method to document the chat_template behavior. No changes to `generate()` or `dispose()`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> HuggingFace Hub | Fetch of chat_template.jinja file over HTTPS |
| LLM output -> Response parser | Untrusted model output parsed into weight objects |
| IndexedDB -> Application | Persisted data loaded on startup could be corrupted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | chat_template.jinja content | accept | Template is fetched from official HuggingFace Hub CDN over HTTPS; the Jinja parser in @huggingface/jinja handles rendering safely. Risk: a compromised HF CDN could serve malicious template, but this is the same trust boundary as the model weights themselves. |
| T-01-02 | Tampering | LLM response output | mitigate | responseParser.js clamps all weights to [-1.0, 1.0], confidence to [0.0, 1.0], validates intent against enum ['exploring', 'deciding', 'focused'], and falls back to previous weights on parse failure. |
| T-01-03 | Denial of Service | chat_template fetch failure | mitigate | On fetch failure (network error, 404, non-200), error propagates through worker to useModelLoader which retries MAX_RETRIES=2 times, then sets status='error' disabling AI (D-04). No infinite retry loops. |
| T-01-04 | Information Disclosure | User browsing data in prompt | accept | All data stays local (local-first architecture). LLM runs in-browser. No data leaves the device. The prompt contains user interaction patterns but these never traverse a network boundary. |
| T-01-05 | Tampering | IndexedDB persisted weights | accept | Weights loaded from IndexedDB on startup could have NaN or out-of-range values from prior corruption. Current mitigation: responseParser clamps on parse, but getWeights() does not re-validate on load. Accepted for PoC scope; CONCERNS.md already documents this for Phase 3 fix. |
</threat_model>
